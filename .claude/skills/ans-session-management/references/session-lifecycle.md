# Session Lifecycle

## AgentSession Dataclass

Defined in `server/session_manager.py`. All fields:

```python
@dataclass
class AgentSession:
    session_id: str              # UUID4 string, generated by register_session()
    session_name: str = ""       # Human-readable, auto-generated from directory basename
    client_type: str = "other"   # "claude", "codex", or "other"
    directory: str = ""          # Absolute path to agent's working directory
    voice_name: str | None = None  # Pool-assigned TTS voice (None if not yet assigned)
    spawn_mode: str = "interactive"  # "interactive" or "headless"
    process_pid: int | None = None   # OS PID of the spawned CLI process
    terminal_pid: int | None = None  # OS PID of the terminal emulator
    mcp_session_id: str | None = None  # Streamable HTTP session ID
    status: str = "connected"    # Lifecycle state (see below)
    started_at: float            # Unix timestamp, default time.time()
    last_activity: float         # Unix timestamp, updated on each tool call
    owning_user_id: str = ""     # Discord user ID who spawned this session
    call_session: CallSession | None = None  # Active voice call, or None
    mode: str = "voice"          # "voice" or "message" (text channel mode)
    text_channel_id: int | None = None  # Target text channel for message mode
    message_session: MessageSession | None = None  # Active message session, or None
```

## Status Values

- `"connected"` -- Initial state after registration. Session exists but has not started a call.
- `"working"` -- Set when a call is initiated or continued. Indicates the session is actively using the voice channel.
- `"idle"` -- Set after a call ends. Session is still registered but not in a call.
- `"disconnected"` -- Set during unregistration. Terminal state.

Status transitions:
```
connected --> working  (initiate_call)
working   --> working  (continue_call, speak_to_user -- last_activity updates)
working   --> idle     (end_call)
idle      --> working  (initiate_call again)
*         --> disconnected  (unregister_session)
```

## Registration Flow

`SessionManager.register_session()` performs these steps in order:

1. Generate `session_id` via `uuid.uuid4()`.
2. Auto-generate `session_name` from `directory` basename (or "session" fallback).
3. Call `_unique_name()` to auto-suffix on collision ("myproject" -> "myproject-2").
4. Create the `AgentSession` dataclass instance.
5. Store in `_sessions[session_id]`.
6. Map `mcp_session_id -> session_id` in `_mcp_to_session` (if HTTP transport).
7. Call `VoicePool.assign_voice(session_id, requested_voice)` -- sets `session.voice_name`.
8. Call `Switchboard.register_session(session_id, session_name)` -- creates message queue.
9. Log and return the session.

Parameters:
- `mcp_session_id`: From Streamable HTTP transport. None for stdio mode (single-client).
- `session_name`: Optional override. Usually auto-derived.
- `client_type`: Determined by spawn detection or init handshake.
- `directory`: The agent's CWD. Used for session naming and session browser matching.
- `spawn_mode`: Set by the spawn module when launching new agents.
- `process_pid`, `terminal_pid`: Set by spawn. Used by `/kill` command.
- `owning_user_id`: Discord user who triggered `/spawn`. Used for permission checks.
- `requested_voice`: Optional explicit voice preference from spawn command.

## Unregistration Flow

`SessionManager.unregister_session(session_id)`:

1. Call `VoicePool.release_voice(session_id)` -- frees the voice for reassignment.
2. Call `Switchboard.unregister_session(session_id)` -- drains and removes the message queue.
3. Pop from `_sessions` dict. Log warning if not found.
4. Remove from `_mcp_to_session` mapping.
5. Decrement the name counter. Delete counter entry if zero.
6. If session has an active `call_session`, remove it from `CallManager._sessions`.
7. Set `status = "disconnected"`.

## SessionManager Internal State

```python
class SessionManager:
    _call_manager: CallManager           # Voice operations delegate
    _message_manager: MessageManager | None  # Message mode delegate
    _runner: BotRunner                   # Discord bot thread reference
    _sessions: dict[str, AgentSession]   # session_id -> AgentSession
    _mcp_to_session: dict[str, str]      # mcp_session_id -> session_id
    _name_counter: Counter[str]          # base_name -> usage count
    _voice_pool: VoicePool               # Per-session voice assignment
    _switchboard: Switchboard            # Message queuing/routing
```

## Voice Operations Delegation

All operations are routed by session mode:

**Voice mode** (`mode="voice"`):
1. Look up the agent session (by `call_id` for continue/speak/end, by `session_id` for initiate).
2. Resolve the effective voice via `VoicePool.resolve_voice(session_id)`.
3. Delegate to `CallManager` with the resolved voice.
4. Update `AgentSession.last_activity` and `status`.
5. For `initiate_call`: link the resulting `CallSession` to `AgentSession.call_session`.
6. For `end_call`: clear `call_session` and set status to `"idle"`.

**Message mode** (`mode="message"`):
1. Look up the agent session and check `_get_mode_for_call_id()`.
2. Delegate to `MessageManager` (sends text messages, waits for user replies).
3. No TTS/STT — messages sent as text, replies received as text or voice message attachments (decoded via ffmpeg + STT).
4. For `initiate_call`: link the resulting `MessageSession` to `AgentSession.message_session`.
5. Longer reply timeout (300s vs 60s for voice).

The agent-facing MCP tool API is identical for both modes — the agent doesn't know which mode is active.

The `_find_session_by_call_id()` helper does a linear scan over `_sessions` to find which agent session owns a given `call_id`. This is acceptable because the expected session count is small (single-digit concurrent agents).

## Name Collision Handling

`_unique_name(base)` uses `Counter[str]`:
- First time "myproject" is seen: counter becomes 1, returns "myproject".
- Second time: counter becomes 2, returns "myproject-2".
- On unregister: counter decrements. If zero, entry deleted.

This means after the first "myproject" unregisters and a new one registers, it gets "myproject" again (counter was deleted at zero).

## Listing Active Sessions

`list_active_sessions()` returns serializable dicts with:
- All AgentSession fields except `call_session` (not serializable).
- `has_queued_messages` and `queued_message_count` from `Switchboard`.
- ISO 8601 formatted `started_at`.
- Skips sessions with `status == "disconnected"`.

## Correction Pass-through

`SessionManager` exposes `add_correction()` and `list_corrections()` that delegate directly to `CallManager`. These are not session-aware -- corrections are global.

## Relationship to CallManager and MessageManager

`SessionManager` wraps but does not replace `CallManager` or `MessageManager`:
- `CallManager` knows about `CallSession` (voice channel, audio, conversation history).
- `MessageManager` knows about `MessageSession` (text channel, reply futures, conversation history).
- `SessionManager` knows about `AgentSession` (identity, voice assignment, message queue, status, mode).
- `AgentSession.call_session` points to the `CallSession` when a voice call is active.
- `AgentSession.message_session` points to the `MessageSession` when a message session is active.
- The `call_manager` and `message_manager` properties provide direct access when needed.

## MessageManager (`server/message_manager.py`)

Handles text channel communication for message-mode sessions:
- Sends agent messages as Discord text messages
- Waits for user replies via `asyncio.Future` (resolved by bot's `on_message` handler)
- Supports voice message attachments (`.ogg` decoded via ffmpeg, transcribed via STT pipeline)
- Cross-event-loop future resolution via `loop.call_soon_threadsafe()`
- 300-second reply timeout (configurable)
